## 1. API 端點分析

### 1.1 高鐵官網訂票 API（逆向工程）

根據開源專案與技術社群資料，高鐵官網訂票系統 (`irs.thsrc.com.tw`) 的 API 端點大致可分為以下幾類：

| 功能 | 端點（推测） | 說明 |
|------|-------------|------|
| 車次查詢 | `POST /IMINT/` | 查詢特定日期/區間的可用班次 |
| 驗證碼取得 | `GET /IMINT/Captcha` | 取得驗證碼圖片 |
| 訂票請求 | `POST /IMINT/` | 提交訂票請求（含驗證碼、身分證、手機） |
| 訂單查詢 | `GET /IMINT/...` | 查詢已訂票券 |

**請求特徵：**
- 使用 `__VIEWSTATE`、`__EVENTVALIDATION` 等 ASP.NET 表單隱藏欄位
- 需要攜帶 Cookie（含 Session ID）
- 驗證碼與訂票請求綁定（同一 Session）

> ⚠️ 注意：這些端點並未對外公開，逆向工程可能觸犯服務條款。

### 1.2 PTX/TDX 運輸資料平台 API

交通部的 **TDX (Transport Data eXchange)** 平台提供官方的開放 API：

| 類別 | API 名稱 | 說明 |
|------|----------|------|
| 靜態資料 | 高鐵基本資料 | 站點、代碼等基本資訊 |
| 動態資料 | 高鐵每日班表 | 各車次發車/到站時間 |
| 動態資料 | 對號座剩餘位 | 各車次區段剩餘座位（以天為單位） |

**API 申請：**
1. 註冊 TDX 會員：https://tdx.transportdata.tw/
2. 選擇「一般會員」（需 3 天審核）
3. 取得 `APP ID` 與 `APP KEY`

**認證方式：**
```
Authorization: Bearer {ACCESS_TOKEN}
```

**範例請求：**
```
GET https://tdx.transportdata.tw/api/v2/Rail/THSR/DailyTimetable/Today
```

> ✅ TDX API 為官方正式 API，完全合規，僅提供「查詢」功能，無法直接訂票。

### 1.3 驗證碼 API

高鐵官網採用圖像驗證碼：
- 類型：扭曲英文字元 + 干擾線/雜點
- 格式：PNG 圖片（約 140×48 像素）
- Session 綁定：每次請求會產生新的 Captcha ID

## 2. 驗證碼方案評估

### 2.1 高鐵驗證碼特性

| 項目 | 說明 |
|------|------|
| 字元數 | 4-5 個英文字母 |
| 字體 | 扭曲、傾斜、不同大小 |
| 干擾 | 背景雜點、干擾線、上方弧線 |
| 變化 | 每次刷新字符位置、顏色都會變化 |

### 2.2 OCR 解決方案比較

| 方案 | 準確率 | 難度 | 建議 |
|------|--------|------|------|
| Tesseract OCR | ~40-50% | 低 | ❌ 不推薦，效果差 |
| OpenCV 影像前處理 + CNN | ~70-80% | 中 | ✅ 可行，需訓練 |
| 第三方 Captcha 服務 (DeathByCaptcha) | ~85%+ | 低 | ✅ 省時，需付費 |
| 自訓練深度學習模型 | ~90%+ | 高 | ✅ 長期可行 |

### 2.3 技術實現路徑

根據 GitHub 開源專案 `maxmilian/thsrc_captcha`：

1. **影像前處理（OpenCV）**
   - 去噪：`fastNlMeansDenoisingColored()`
   - 去除干擾線：影像二值化 + 形態學運算
   - 字元切割或直接整張輸入

2. **模型訓練（CNN）**
   - 架構：類似 ResNet 的卷積網路
   - 輸入：140×140×3 圖像
   - 輸出：4-5 個字元的分類（26×4 或 26×5）

3. **實測準確率**
   - 社群回報：約 70-80%（視訓練資料量）

### 2.4 重試機制建議

- 驗證碼辨識失敗 → 自動重新請求新的 Captcha
- 訂票請求失敗（如額滿）→ 自動重試（限流）
- 建議間隔：每次請求間隔 1-3 秒，避免被 ban

## 3. 法律與風險

### 3.1 鐵路法第 65 條

> **（105 年 11 月 9 日修正）**
>
> 購買車票加價出售或換取不正利益圖利者，按車票張數，處每張車票價格之 **一倍至十倍** 罰鍰。
>
> 以**不正方法**將虛偽資料或不正指令輸入電腦或其相關設備而購買車票、取得訂票或取票憑證者，處 **五年以下有期徒刑** 或科或併科新臺幣 **三百萬以下** 罰金。

### 3.2 「不正方法」的定義

根據法院實務見解：

| 行為 | 是否觸法？ |
|------|------------|
| 使用別人身分證字號訂票 | ✅ 觸法（影響票務管理正確性） |
| 使用虛擬身分證字號規避停權 | ✅ 觸法 |
| 寫程式自動訂票（無虛偽資料） | ⚠️ 灰色地帶，爭議較大 |
| 使用驗證碼辨識技術 | ⚠️ 涉及「不正方法輸入」之認定 |
| 大量訂票（壟斷車票） | ✅ 觸法（刑法責任） |

### 3.3 風險評估

| 行為 | 法律風險 | 帳號風險 |
|------|----------|----------|
| 官方 TDX API 查詢 | ✅ 無 | ✅ 無 |
| 官網手動訂票（自己帳號） | ✅ 無 | ⚠️ 頻繁失敗可能暫停 |
| 自動化訂票（含驗證碼破解） | ⚠️ 可能觸法 | ✅ 高（IP 被 ban） |
| 黃牛行為（加價轉售） | ❌ 確定觸法 | ❌ 帳號永封 |

### 3.4 合規建議

1. **僅使用官方 API**
   - TDX 平台提供的查詢功能完全合規
   - 如需訂票，透過官方網站/App 手動操作

2. **避免使用驗證碼破解**
   - 涉及「不正方法輸入」爭議
   - 103 年後實務見解可能有不同認定

3. **公益用途例外**
   - 如為身心障礙者設計輔助購票工具，理論上可主張正当理由
   - 但仍需謹慎，建議諮詢律師

4. **如要研究**
   - 僅用於學術/技術學習目的
   - 不要公開發布 / 商業化
   - 嚴格控制在最小範圍內

## 4. 結論與建議

### 4.1 技術可行性

| 功能 | 可行性 | 備註 |
|------|--------|------|
| 查詢班次（TDX API） | ✅ 完全可行 | 官方 API，完全合規 |
| 查詢剩餘座位 | ✅ 完全可行 | TDX 有提供 |
| 自動化訂票 | ⚠️ 技術可行 | 但涉及法律風險 |
| 驗證碼破解 | ⚠️ 可實現 | CNN 準確率約 70-80% |

### 4.2 推薦路徑

1. **合法合規方案**
   - 使用 TDX API 做班次查詢
   - 透過官方 App 完成訂票（可使用快捷指令自動化部分流程）

2. **研究學習方案**
   - 如僅想學習 Captcha 破解技術，可使用開源專案練習
   - **千萬不要**對外公開發布或提供服務

### 4.3 風險提醒

> 🚨 本報告僅供技術調研參考。任何自動化訂票行為均可能觸犯鐵路法第 65 條，請勿以身試法。如有實際需求，建議透過正規管道購票，或諮詢專業法律意見。
